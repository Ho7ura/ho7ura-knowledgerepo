
跨核有问题的只有这 4 个：

变量	写者 (核心)	读者 (核心)	后果
aud_ws_ready	        loop() WS回调 (Core 1)	mic_cap (Core 0)	Core 0 看不到 WS 断开，一直空采
run_audio_stream	loop() WS回调 (Core 1)	mic_cap (Core 0)	关不掉音频流
http_play_running	stopStreamWav() (Core 1)	taskHttpPlay (Core 0)	停不掉喇叭播放
tts_playing	                外部写入 (大概率 Core 1)	taskTTSPlay (Core 0)	开关 TTS 不及时生效

硬件缓存不互通
M3/M4 的 volatile 防编译器优化就够，因为单核没有 cache 一致性问题。

ESP32 每个核有自己的 L1 Cache，像两个独立的小本子：


Core 0 内存视图:                  Core 1 内存视图:
┌──────────────┐                ┌──────────────┐
│ L1 Cache     │                │ L1 Cache     │
│ run_audio = ?│                │ run_audio = 1│  ← 还是旧的
└──────┬───────┘                └──────┬───────┘
       │                               │
       └──────────┬────────────────────┘
                  │
           ┌──────┴───────┐
           │  SRAM (真实值)│
           │ run_audio = 0│  ← Core 1 写入在这里
           └──────────────┘
Core 1 写了 run_audio_stream = false：

数据先到 Core 1 的 store buffer → 然后才刷到 Core 1 的 L1 cache → 最终才到 SRAM
Core 0 不会自动知道 SRAM 变了，它可能一直读自己 L1 cache 里的旧值
volatile 只能让编译器每次生成 load/store 指令，不能强制刷新硬件 cache。

这就是跨核不可见：Core 1 写了一件事，Core 0 可能永远看不到。
