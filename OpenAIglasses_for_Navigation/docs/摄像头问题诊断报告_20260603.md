# 摄像头问题诊断报告（2026-06-03）

> 硬件：Seeed Studio XIAO ESP32S3 Sense + OV3660 摄像头模块（新换）
> 固件：compile/src/main.cpp (PlatformIO, Arduino framework)

## 现象

新 OV3660 模块和旧模块症状完全一致：

```
[WiFi] connecting.... OK 172.20.10.2
E (1696) camera: Detected camera not supported.
E (1697) camera: Camera probe failed with error 0x106(ESP_ERR_NOT_SUPPORTED)
[CAM] init failed, reboot...
```

无限重启循环。

## 错误码含义

`0x106 = ESP_ERR_NOT_SUPPORTED`

`esp_camera_init()` 内部流程：
1. `camera_enable_out_clock` — 开启 XCLK（GPIO10, 20MHz）
2. `SCCB_Init(40, 39)` — 初始化摄像头 I2C 总线（GPIO40=SDA, GPIO39=SCL）
3. `SCCB_Probe()` — 扫描已知传感器地址，**找到了设备**（有 ACK 应答）
4. 遍历 `g_sensors[]` 中各传感器 detect 函数 → **全部失败**
5. 返回 `ESP_ERR_NOT_SUPPORTED`

如果 SCCB_Probe 找不到任何设备，错误码是 `ESP_ERR_NOT_FOUND`。现在是 `ESP_ERR_NOT_SUPPORTED`，说明 I2C 总线上有设备应答，但 PID 寄存器值与所有已知型号都不匹配。

## OV3660 detect 函数分析

库源码中 `ov3660_detect()` 的逻辑：

```c
int ov3660_detect(int slv_addr, sensor_id_t *id) {
    if (OV3660_SCCB_ADDR == slv_addr || slv_addr == 0x30) {
        uint8_t h = SCCB_Read16(slv_addr, 0x300A);
        uint8_t l = SCCB_Read16(slv_addr, 0x300B);
        uint16_t PID = (h<<8) | l;
        if (OV3660_PID == PID) {  // OV3660_PID = 0x3660
            return PID;
        }
    }
    return 0;
}
```

条件：只在 I2C 地址为 `0x3C` (OV3660_SCCB_ADDR) 或 `0x30` 时才检测。
失败原因：传感器 PID 寄存器 `0x300A:0x300B` 读出的值 ≠ `0x3660`。

## 诊断过程

### 尝试 1：库自带的 SCCB 函数做"事后诊断"（失败→crash）

`esp_camera_init` 失败后会释放 I2C 总线，之后再调用 `SCCB_Probe()`/`SCCB_Read16()` 导致空指针 panic。

### 尝试 2：Arduino Wire + ledcAttach 做"事前诊断"（找不到设备）

用 `ledcAttach` 开 XCLK + `Wire.begin` 扫描 I2C → **找不到任何设备**。原因是 Arduino LEDC API 与库内部 `camera_enable_out_clock` 的底层实现不一致（缺少 `periph_ll_enable_clk_clear_rst`），XCLK 未正确产生。

### 尝试 3：ESP-IDF ledc_timer_config + 库 SCCB 函数（找不到设备）

完全复刻库内部的 LEDC + SCCB 设置，但调用的 `SCCB_Probe` 返回 `0x105` (NOT_FOUND)。原因：库的预编译 `.a` 文件使用新版 **sccb-ng** 驱动（基于 new I2C master driver），而 `extern "C"` 声明链接到旧版 SCCB 函数（基于 legacy I2C driver），两者行为不同。

### 尝试 4：强制从源码编译（链接失败）

删除预编译 `.a` 后，PlatformIO 不会自动用 `lib_deps` 中的源码替代，链接时报 `cannot find -lespressif__esp32-camera`。

## 总结

| 项目 | 结果 |
|------|------|
| WiFi / ESP32 硬件 | 正常 |
| I2C 物理连接 | 有 ACK 应答（库内 SCCB_Probe 找到了设备） |
| 传感器 PID | ≠ 0x3660（全部 detect 函数匹配失败） |
| 两个 OV3660 模块 | 表现完全一致 |
| 外部诊断局限 | 无法复现库内部的 sccb-ng I2C 驱动行为 |

两个 OV3660 模块都能在 I2C 上 ACK，但返回的 PID 不为 0x3660。不是 FPC 接触不良（接触不良不会有稳定的 ACK），更可能是模块型号不对或兼容性问题。

## 建议下一步

1. **用确定完好的 OV2640 模块交叉验证** — XIAO ESP32S3 Sense 官方默认配 OV2640。如果 OV2640 正常工作，则板子和 FPC 座都没问题，问题出在这两个 OV3660 模块上
2. **确认 OV3660 模块来源** — 两个模块是否是同一批次/同一卖家？是否有可能是标签错误的非 OV3660
3. **PC 摄像头临时方案** — 用 `local_cam.py` + PC USB 摄像头推流到服务器，绕过 ESP32 摄像头问题继续开发 AI 导航功能
